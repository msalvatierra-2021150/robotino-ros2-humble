# Robotino 3 ROS 2 Command Reference

## 1. Source ROS 2 Humble

Run this in every new terminal before using ROS 2 commands:

```bash
source /opt/ros/humble/setup.bash
```

## 2. Source the Robotino workspace

Run this after sourcing ROS 2, so the terminal can find the Robotino packages:

```bash
source ~/robotino3/robotino3ros2/install/setup.bash
```

---

# Real Robotino Commands

## 3. Connect to the Robotino base

Launch the basic Robotino bringup node and connect to the Robotino controller:

```bash
ros2 launch rto_bringup rto_bringup_launch.py hostname:=172.27.1.1
```

Use this when connecting directly to the Robotino using the `172.27.1.1` address.

## 4. Move the Robotino through `/rto3/cmd_vel`

Send a small forward velocity command to the Robotino for 10 seconds:

```bash
timeout 10s ros2 topic pub /rto3/cmd_vel geometry_msgs/msg/Twist "{linear: {x: 0.05, y: 0.0, z: 0.0}, angular: {x: 0.0, y: 0.0, z: 0.0}}" -r 1
```

This is useful for testing whether the Robotino can receive velocity commands.

## 5. Launch odometry, sensors, RViz, and robot description

Launch the Robotino odometry, sensors, RViz visualization, and URDF/XACRO robot description publisher:

```bash
ros2 launch rto_bringup rto_odom_sensors_rviz.launch.py hostname:=172.27.1.1
```

Use this to visualize the robot model, sensor data, and odometry in RViz.

## 6. Publish velocity commands to `/cmd_vel`

Send a velocity command to the `/cmd_vel` topic:

```bash
ros2 topic pub /cmd_vel geometry_msgs/msg/Twist \
"{linear: {x: 0.2}, angular: {z: 0.0}}"
```

Use this when your stack expects commands on `/cmd_vel` instead of `/rto3/cmd_vel`.

# Gazebo Simulation Commands

## 7. Generate the Robotino URDF from XACRO

Convert the Robotino XACRO file into a URDF file:

```bash
ros2 run xacro xacro rto_description/urdf/robots/rto-3.urdf.xacro > /tmp/rto-3.urdf
```

## 8. Spawn the Robotino model in Gazebo

Publish the generated URDF model into Gazebo:

```bash
ros2 run gazebo_ros spawn_entity.py -entity robotino -file /tmp/rto-3.urdf
```

## 9. Launch the Gazebo simulation

Start the Robotino Gazebo simulation environment:

```bash
ros2 launch rto_simulation gazebo_model.launch.py
```

# Mapping and SLAM

## 10. Save a map generated by SLAM Toolbox

Use this after SLAM Toolbox has created a map:

```bash
ros2 run nav2_map_server map_saver_cli -f ~/robotino3/maps/robotino_map
```

This saves the current map as:

```bash
~/robotino3/maps/robotino_map.yaml
~/robotino3/maps/robotino_map.pgm
```

# Nav2 Commands

## 11. Run Nav2 with a custom YAML config

For simulation, use:

```bash
ros2 launch nav2_bringup navigation_launch.py \
  use_sim_time:=true \
  params_file:=/home/mike/robotino3/robotino3ros2/rto_simulation/config/nav2_params_robotino.yaml
```

For the real Robotino, `use_sim_time` should normally be `false`.

# Frontier Exploration

## 12. Run the frontier exploration nodes

Launch the classical frontier exploration package:

```bash
ros2 launch frontier_exploration classical_exploration.launch.py
```

This starts the frontier detector and the frontier navigation node.

## 13. Request a frontier goal manually

Call the frontier service and request the best frontier goal:

```bash
ros2 service call /frontier_pose frontier_interfaces/srv/FrontierGoal "{goal_rank: 0}"
```

Use `goal_rank: 0` for the highest-ranked frontier.

# Raspberry Pi / LiDAR

## 14. Connect to the Raspberry Pi 4 through SSH

SSH into the Raspberry Pi that runs the LiDAR:

```bash
ssh robotino@192.168.0.100
```

Use this to start, stop, or debug the LiDAR node.

# Real Robotino Autonomous Exploration

## 15. Connect to the real Robotino and start mapping/navigation

Launch the real exploration stack without frontier exploration:

```bash
ros2 launch rto_bringup real_exploration.launch.py start_frontier:=false autostart:=false robot_ip:=192.168.0.20
```

Use this first to test SLAM, Nav2, odometry, TF, and LiDAR before enabling frontier navigation.

## 16. Launch real exploration with frontier enabled

Once Nav2, SLAM, TF, and LiDAR timestamps are working correctly, run:

```bash
ros2 launch rto_bringup real_exploration.launch.py start_frontier:=true robot_ip:=192.168.0.20
```

This launches the real Robotino exploration stack and starts frontier exploration after the map becomes available.

# Time Synchronization

## 17. Important: synchronize Raspberry Pi and laptop time

The Raspberry Pi and laptop must have synchronized clocks. If their clocks differ, `/scan` timestamps may be several seconds behind the laptop clock, causing errors such as:

```text
Transform data too old
Message Filter dropping message
Unable to transform robot pose
```

The Raspberry Pi should sync to the laptop at boot, and the laptop must have the time server running.

On the laptop, check and restart Chrony:

```bash
sudo systemctl restart chrony
sudo systemctl enable chrony
chronyc sources -v
chronyc tracking
```

On the Raspberry Pi, check that it is syncing to the laptop:

```bash
chronyc sources -v
chronyc tracking
```

The Pi should show the laptop IP as its selected time source, usually marked with `^*`.

## 18. Check laptop and Raspberry Pi time

From the laptop, compare both clocks:

```bash
echo "Laptop:"
date +%s.%N

echo "Pi:"
ssh robotino@192.168.0.100 "date +%s.%N"
```

The timestamps should be very close. A difference of several seconds will break SLAM/Nav2 timing.

## 19. Check LiDAR timestamp against laptop time

After starting the LiDAR, verify that `/scan` timestamps are close to the laptop clock:

```bash
ros2 topic echo /scan --once --field header
date +%s.%N
```

The `/scan` timestamp should not be 7–9 seconds behind the laptop time.

# Useful Debugging Commands

## Check active ROS 2 nodes

```bash
ros2 node list
```

## Check Nav2 lifecycle state

```bash
ros2 lifecycle get /controller_server
ros2 lifecycle get /planner_server
ros2 lifecycle get /bt_navigator
```

You want them to be:

```text
active
```

## Check Nav2 action server

```bash
ros2 action list | grep navigate
```

You should see:

```text
/navigate_to_pose
```

## Check odometry

```bash
ros2 topic hz /odom
ros2 topic echo /odom --once
```

## Check LiDAR

```bash
ros2 topic hz /scan
ros2 topic echo /scan --once --field header
```

## Check TF between odometry and robot base

```bash
ros2 run tf2_ros tf2_echo odom base_link
```

## Check TF between map and odom

```bash
ros2 run tf2_ros tf2_monitor map odom
```

Use this to detect stale `map -> odom` transforms.

## Teleoperation, AprilTags, and Map Saving

### Drive the Robotino

```bash
ros2 run teleop_twist_keyboard teleop_twist_keyboard
```

Use the keyboard to manually move the robot. The terminal must stay selected.

### Check AprilTag detections

```bash
ros2 topic echo /detections
```

This shows raw AprilTag detections, including the tag ID and estimated pose. No output appears when no tag is visible.

### Start the eMDB bridge

```bash
ros2 run rto_emdb_bridge apriltag_to_emdb_bridge
```

This converts `/detections` messages into the format used by the eMDB system.

Check the bridge output topic with:

```bash
ros2 node info /apriltag_to_emdb_bridge
```

Then echo the published eMDB topic, for example:

```bash
ros2 topic echo /robotino/emdb/tag_detection
```

### Save the map

Create a folder:

```bash
mkdir -p /home/mike/robotino_maps
```

Save the current map:

```bash
ros2 run nav2_map_server map_saver_cli \
  -f /home/mike/robotino_maps/robotino_lab
```

This normally creates:

```text
robotino_lab.yaml
robotino_lab.pgm
```

Confirm the files:

```bash
ls -lh /home/mike/robotino_maps/
```

### Launch with the saved map

```bash
ros2 launch rto_simulation gazebo_model.launch.py \
  use_saved_map:=true \
  map_yaml_file:=/home/mike/robotino_maps/robotino_lab.yaml
```

This loads the saved map instead of starting a new mapping session.
